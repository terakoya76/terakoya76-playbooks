# Business Email Review

ビジネスメールの添削を行う。原文の意図を保ったまま、相手に誠実かつ明確に伝わる表現に整える。指摘は「直すべき理由」とセットで示し、相手の学習につながるようにする。

## 添削ワークフロー

以下の順に実行する。

### 1. 前提情報の確認

メール本文だけが渡された場合、必要に応じて以下を確認する（不明なまま推測しない）。

- 送信者と受信者の関係（社内/社外、上下関係、初回接触かどうか）
- 目的（依頼・謝罪・報告・提案・催促など）
- 言語（日本語 / 英語 / 併記）
- フォーマリティの希望（堅め / 標準 / カジュアル寄り）

ユーザーが明示していない場合は、推定した前提を冒頭に明示した上で添削を進める。ズレていれば指摘してもらう。

### 2. 構造レビュー

ビジネスメールの基本構成に沿っているかを確認する。

- 件名：用件が 20 文字以内で伝わるか、相手が受信箱で判別できるか
- 宛名：会社名・部署・役職・氏名の順、敬称（様 / 御中）の正しさ
- 挨拶：関係性に応じた冒頭文（お世話になっております／初めてご連絡いたします 等）
- 名乗り：社名・部署・氏名
- 本題：結論ファースト、背景・詳細・依頼事項の順
- 締め：次アクション、返信期限、連絡先
- 署名

欠けている要素、順序が崩れている箇所を指摘する。

### 3. 表現レビュー

以下の観点で問題箇所を洗い出す。

- **敬語の正確さ**：二重敬語、尊敬語/謙譲語の取り違え、バイト敬語（「〜のほう」「〜になります」誤用）
- **トーンの一貫性**：丁寧体と常体の混在、過度な謙遜や過度な断定
- **曖昧さ**：「なるはや」「別途」「追って」など期限や主語が不明確な表現
- **冗長さ**：同じ意味の繰り返し、前置きの長さ、クッション言葉の過多
- **依頼の明確さ**：何を・いつまでに・どの形式で、が揃っているか
- **誤字脱字・変換ミス**
- **機密配慮**：CC/BCC の使い分けが本文から読み取れるか、社外秘に触れていないか

英語メールの場合は、以下も確認する。

- 件名の動詞始め / 名詞句の適切さ
- 挨拶（Dear / Hi / Hello）と締め（Best regards / Kind regards / Sincerely）の整合
- 時制、冠詞、前置詞
- 直訳調（"I think that..." の連発など）の回避
- アクション依頼の明示（Could you / Would you / Please）

### 4. 出力フォーマット

以下の構成で返す。マークダウンで整形する。

```
## 前提（推定）
- 関係性: ...
- 目的: ...
- フォーマリティ: ...

## 修正版
（添削後の完成メール全文）

## 主な修正点
1. **[箇所]** 原文「...」→ 修正「...」
   理由: ...
2. ...

## さらに検討したい点
- （任意の補足・選択肢提示）
```

「修正版」は、そのままコピペで送信できる状態にする。「主な修正点」は 3〜7 件程度に絞り、些末な句読点よりも伝達に影響する変更を優先する。

### 5. 原文保持の原則

- 元の意図・主張を変えない。内容の正誤は判断しない（事実関係に疑義がある場合のみ「確認ポイント」として提示）
- 過度に硬くしない。相手との関係性に不自然な敬度を足さない
- 個人の文体のクセ（一人称、定型表現）は、ビジネス上問題なければ残す

## 頻出の直しパターン

参考として、日本語ビジネスメールでよく直す型を挙げる。

- 「ご確認させていただきます」→「確認いたします」（二重敬語）
- 「〜させていただきます」の多用 → 動作が相手の許可を要するときのみ使う
- 「取り急ぎ」→ 用件が急ぎの理由を一行添えるか削除
- 「〜のほう」「〜になります」→ 指示対象を明確に
- 「お手数おかけしますが」の連発 → 一通に一度まで
- 結論が最後に来ている → 冒頭に用件サマリを置く

## してはいけないこと

- ユーザーが明示していない事実を本文に足さない（日時・金額・固有名詞など）
- 機密情報の可能性がある内容を外部ツールに転送しない
- 原文の論旨を書き換えない。迷ったらユーザーに確認する
- 「完璧です」など中身のない評価で終わらない。必ず 1 箇所以上、改善余地または確認ポイントを挙げる
